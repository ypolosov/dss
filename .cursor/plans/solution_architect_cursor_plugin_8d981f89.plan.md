# План: Cursor Plugin «Solution Architect» (ADD 3.0 + wiki-документация)

## Цели и границы

- **Плагин**: Cursor plugin с правилами, навыками и (при необходимости) командами/агентами.
- **Режим**: человек ведёт, агент помогает — пошаговые сценарии по ADD 3.0 с подтверждением пользователя.
- **Документация**: в репозитории целевого проекта; Markdown + привязка к Structurizr (`!docs`, `!adrs`) для единого workspace.
- **Диаграммы**: в плане — сравнительная таблица и рекомендация (Structurizr vs Mermaid vs mingrammer/diagrams); реализация — по выбранной рекомендации.

---

## 1. Методология ADD 3.0 — что кодировать в плагине

ADD 3.0 (SEI) — итеративный процесс с двумя частями.

**Часть 1 — тактики и качество:**

1. Проверить достаточность требований.
2. Выбрать элемент системы для декомпозиции.
3. Выявить кандидатов архитектурных драйверов (функциональные, качественные атрибуты, ограничения, тип системы, цели, concerns).
4. Выбрать дизайн-концепт (паттерн, тактика, reference architecture, framework), удовлетворяющий драйверам.

**Часть 2 — документирование декомпозиции:**

1. Создать экземпляры элементов и распределить ответственности.
2. Определить интерфейсы.
3. Верифицировать и уточнить требования, зафиксировать ограничения для элементов.
4. Повторить для следующего элемента.

**Артефакты для wiki:** драйверы (зафиксированные требования), выбранные элементы и ответственности, интерфейсы, ADR по ключевым решениям, представления (views) — диаграммы C4/деплой.

Плагин должен направлять пользователя по этим шагам и помогать заполнять/обновлять документы в выбранной структуре (см. ниже).

## 2. Структура документации в репозитории (Structurizr + Markdown)

Рекомендуемая структура в целевом проекте (например `docs/architecture/` или корень `.structurizr/`):

- **workspace.dsl** — модель C4 и ссылки на документы:
  - `!docs <path>` — каталог с Markdown (описание контекста, компонентов, решений).
  - `!adrs <path>` — каталог с ADR (MADR или adr-tools формат).
- **docs/** (или `docs/architecture/`) — Markdown-файлы: драйверы, контекст, глоссарий, решения (при необходимости дублирующие ADR для удобства чтения).
- **adrs/** — ADR в формате MADR или adr-tools (совместимо с Structurizr импортерами).

Плагин должен:

- Предлагать/создавать эту структуру при инициализации «архитектуры» в проекте.
- В навыках (skills) описывать, куда что писать (какой файл/раздел под какой шаг ADD).
- Правила (rules) — подсказывать агенту использовать эту структуру и шаги ADD при ответах про архитектуру.

## 3. Инструменты для диаграмм — сравнение и рекомендация


| Критерий         | Structurizr (DSL + local/export)                                                                       | Mermaid (C4)                              | mingrammer/diagrams                         |
| ---------------- | ------------------------------------------------------------------------------------------------------ | ----------------------------------------- | ------------------------------------------- |
| **Модель**       | C4, один источник правды, связи с !docs/!adrs                                                          | C4 (experimental), отдельные диаграммы    | Нет C4, инфра/облако (AWS, GCP, K8s и т.д.) |
| **Формат**       | Текстовый DSL, версионируемый                                                                          | Текстовый, в Markdown                     | Python-код, версионируемый                  |
| **Просмотр**     | Structurizr local (vNext) или export → Mermaid/PlantUML/HTML                                           | Встроено в GitHub/GitLab, VS Code, Cursor | PNG/SVG из скрипта                          |
| **Деплой/инфра** | Через C4 Deployment/включения                                                                          | Ограниченно                               | Сильная сторона                             |
| **Поддержка**    | Lite deprecated, актуально: [Structurizr vNext](https://docs.structurizr.com/commands) (local, export) | Активная, C4 встроен                      | Активная                                    |
| **Зависимости**  | Java/Node + Structurizr CLI или Docker                                                                 | Нет (или рендерер в браузере)             | Python + Graphviz                           |


**Рекомендация для плагина:**

- **Основной путь (C4 + wiki):** Structurizr DSL в репозитории как единый источник правды для софтверной архитектуры и связей с документацией; просмотр/редактирование через **Structurizr vNext** (`local` — аналог Lite) и/или **export** в Mermaid/PlantUML для встраивания в Markdown.
- **Упрощённый вариант (без Java/Docker):** если не хочется поднимать Structurizr local — вести C4 в **Mermaid** (синтаксис C4) прямо в Markdown; плагин может содержать шаблоны Mermaid C4 под контекст/контейнер/компонент.
- **mingrammer/diagrams:** рекомендовать как **дополнение** для диаграмм развёртывания/инфра (облако, K8s), а не замену C4 для логической архитектуры; в плагине — опциональный навык или раздел «инфра-диаграммы».

Итог: в реализации плагина заложить поддержку **Structurizr-first** (структура workspace.dsl + !docs + !adrs), плюс краткие инструкции/шаблоны для Mermaid C4 и (опционально) для mingrammer/diagrams в качестве второго яруса.

## 4. Компоненты плагина (Cursor Plugin)

Использовать [create-plugin scaffold](.cursor/plugins/.../create-plugin-scaffold/SKILL.md): один плагин, минимальный набор компонентов.

- **Rules (rules/):**
  - При обсуждении архитектуры целевого проекта — следовать ADD 3.0 (шаги, драйверы, декомпозиция, документирование).
  - Использовать и предлагать структуру документации в репозитории: `workspace.dsl`, `!docs`, `!adrs`, каталоги docs/ и adrs/.
- **Skills (skills/):**
  - **ADD 3.0 step-by-step** — пошаговый сценарий: какой шаг ADD, какие артефакты создавать/обновлять, где в репозитории (файлы, разделы).
  - **Init architecture docs** — инициализация структуры (workspace.dsl с заглушками, папки docs/, adrs/, пример одного ADR и одного doc).
  - **Structurizr DSL quick ref** — краткая шпаргалка по DSL (C4, !docs, !adrs) и командам vNext (local, export).
  - Опционально: **Diagrams** — когда использовать Structurizr export vs Mermaid C4 vs mingrammer (инфра).
- **Agents (agents/):** опционально один агент «solution-architect» с описанием: помощь по ADD 3.0 и ведению wiki-документации в стиле выше; явно указать режим «human-guided».
- **Commands (commands/):** опционально команды вида «Initialize architecture folder» или «Start ADD iteration» — вызов агента/навыка с подсказкой шага.

Реализация MCP не обязательна для первой версии: достаточно правил и навыков, чтобы агент в Composer выполнял шаги и писал в файлы; при необходимости позже можно добавить MCP для вызова Structurizr export/validate.

## 5. Зависимости и окружение (для пользователя плагина)

- **Structurizr:** установка [Structurizr vNext](https://docs.structurizr.com/commands) (local, export) — по официальной инструкции (например, Java + JAR или будущий бинарник); либо Docker. В плагине — ссылка на docs и проверка наличия `workspace.dsl` в ожидаемой директории.
- **Просмотр без Structurizr:** export в Mermaid → просмотр в Cursor/IDE или в CI (например, Mermaid in Markdown).
- **diagrams (mingrammer):** опционально Python + `diagrams` + Graphviz; в плагине — только описание в навыке, без обязательной установки.

## 6. Порядок реализации (кратко)

1. Создать каркас плагина (manifest, rules, skills, при необходимости agent + command).
2. Описать в правилах привязку к ADD 3.0 и к структуре docs + workspace.dsl + !docs/!adrs.
3. Реализовать навык «Init architecture docs» (шаблоны workspace.dsl, папки, пример doc и ADR).
4. Реализовать навык «ADD 3.0 step-by-step» с привязкой шагов к артефактам и файлам.
5. Добавить навык «Structurizr DSL quick ref» и (опционально) «Diagrams» с рекомендацией из п.3.
6. При необходимости — агент + команда для пошагового запуска.
7. README плагина: назначение, ADD 3.0, структура репозитория, установка Structurizr vNext, варианты диаграмм (Structurizr / Mermaid / diagrams).

После этого план можно считать достаточным для начала реализации; детали формата MADR vs adr-tools и точные имена файлов можно зафиксировать на этапе реализации (например, MADR по умолчанию).
